# 3. 变更评估

[返回工作流框架](../../AI自动化研发工作流实践指南.md)

## 产物文件

读取当前任务的需求解析或澄清文件，写入 `/Users/j/codes/ai-workflow/tasks/<task-id>/03-change-assessment.json`，并更新 `task.yaml`。

## 目标

由工作流执行器依据需求和代码扫描结果自动选择执行路径。资料不足时先退回澄清，不将不确定需求直接归为复杂需求。

## 适用路径

- 快速修复、轻量需求、复杂需求均必须经过本节点。
- `NEEDS_CLARIFICATION` 不是执行路径，而是退回需求澄清的阻断状态。

## 路径级准入产物

| 路径 | 必需产物 |
| --- | --- |
| 快速修复 | 需求解析包、目标模块扫描结果 |
| 轻量需求 | 需求解析包、目标模块扫描结果 |
| 复杂需求 | 需求解析包、目标模块扫描结果 |

快速修复的评估结果必须额外生成执行摘要，至少说明当前行为、预期行为、改动范围、验证方式和回滚方式。该摘要是快速修复进入开发实现的直接准入产物。

## 输入

- [需求受理](./01-requirement-intake.md)或[需求澄清](./02-requirement-clarification.md)输出的需求解析包。
- 目标模块的历史问题、线上监控和公共依赖信息。
- 目标模块的路由、接口、权限、状态和公共依赖扫描结果。

## 执行

使用 `$03-change-assessment`，工作流执行器按以下顺序判断：

1. **资料完整性检查**：确认需求类型、目标模块、当前行为、预期行为，以及相关设计或接口信息是否足够判断。
2. **复杂触发器检查**：扫描是否涉及高风险业务、破坏性变更、公共能力或跨业务模块影响。
3. **快速修复检查**：确认是否为恢复既有行为的局部缺陷，且不存在复杂触发器。
4. **轻量需求检查**：确认是否为资料明确、单业务模块内的常规功能或体验调整。
5. **低置信度处理**：资料虽完整但扫描结果冲突或无法判断时，保留自动建议并等待 Owner 确认。

AI 只负责从需求和代码中提取判定信号；最终路径由固定规则引擎决定，不允许模型仅凭主观判断直接分类。规则优先级固定为：`NEEDS_CLARIFICATION -> 复杂需求 -> 快速修复 -> 轻量需求`。不允许通过手工选择绕过自动判定。

## 自动判定规则

| 判定结果 | 自动规则 |
| --- | --- |
| `NEEDS_CLARIFICATION` | 缺少需求类型、目标模块、预期行为、关键接口或设计信息；或资料之间存在冲突。此状态不是路径，必须退回需求澄清。 |
| 复杂需求 | 命中任一硬触发器：权限、安全、金额、隐私、不可逆删除、数据迁移、破坏性接口契约、公共组件/公共包、全局样式/配置/路由、跨业务模块改动；或存在多个独立外部依赖、回滚风险。 |
| 快速修复 | 必须同时满足：修复已有且已定义的行为；问题和预期可明确复现；影响局限于一个业务模块；不新增功能、不改接口契约、不改数据或权限、不改公共能力或跨模块状态；具备针对性验证和快速回滚方式。 |
| 轻量需求 | 资料与设计明确；影响局限于一个业务模块；使用已有接口，或使用已确认且向后兼容的模块专用接口；仅调整本地页面、组件、交互或状态；未命中复杂触发器。 |

“业务模块”按路由归属、目录边界、Owner 和公共依赖共同识别，不以单纯文件数量作为判定依据。

## 本节点产物

- `change-assessment.json`，至少包含：

```json
{
  "decisionState": "ROUTED | NEEDS_CLARIFICATION",
  "path": "quick_fix | lightweight | complex | null",
  "confidence": "high | medium | low",
  "matchedRules": [],
  "evidence": [],
  "complexTriggers": [],
  "requiredNodes": [],
  "requiredArtifacts": {},
  "executionBrief": {},
  "requiresOwnerApproval": false
}
```

- 风险说明、需要人工确认的项目和不可跳过的节点。

`requiredArtifacts` 必须按判定路径列出后续节点的准入产物：快速修复使用 `executionBrief`；轻量需求使用需求解析包和轻量实施计划；复杂需求使用验收清单、影响范围报告和已批准实施计划。

## 完成条件

- 已生成结构合法的 `change-assessment.json`。
- 每条命中规则都有可追溯的需求片段、代码位置或扫描结果作为 `evidence`。
- `decisionState=ROUTED` 时，`requiredNodes` 与判定路径一致，工作流执行器已据此锁定或解锁节点。
- `decisionState=NEEDS_CLARIFICATION` 时，任务已退回需求澄清且未解锁开发实现。

## 执行门禁

- 未生成 `change-assessment.json` 时，开发实现节点状态必须为 `BLOCKED`。
- `decisionState=NEEDS_CLARIFICATION` 时，任务进入 `BLOCKED`，只能返回需求澄清补齐资料。
- `path=complex` 且 `confidence=high` 时，自动解锁复杂需求路径；方案评审节点负责正式风险确认。
- `confidence` 不为 `high` 或人工需要覆盖自动结论时，任务进入 `WAITING_APPROVAL`。
- 人工需要覆盖自动结论时，必须记录覆盖路径、原因、审批人和时间。

## 缺失处理

- 快速修复缺少可验证的当前行为、预期行为或回滚方式时，改为 `NEEDS_CLARIFICATION`。
- 轻量需求或复杂需求缺少需求解析包时，退回 [需求澄清](./02-requirement-clarification.md)。
- 扫描结果无法判断是否命中复杂触发器时，标记为低置信度并等待 Owner 确认，不得直接进入开发。

## 人工关口

高风险或判断不确定时，由模块 Owner 确认等级。

## 交接

- 复杂需求进入 [验收设计](./04-acceptance-design.md)。
- 轻量需求在需求澄清中补充最小验收条件后，进入 [实施设计](./06-implementation-design.md)。
- 快速修复直接进入 [开发实现](./08-development-implementation.md)。
