# 11. 发布管理

[返回工作流框架](../../AI自动化研发工作流实践指南.md)

## 产物文件

读取发布准入产物，写入 `/Users/j/codes/ai-workflow/tasks/<task-id>/11-release-record.md`，并更新 `task.yaml`。

## 目标

以可控、可观察、可回滚的方式将已批准变更发布到目标环境。

## 适用路径

- 复杂需求独立经过本节点。
- 快速修复和轻量需求默认由所属发布批次统一经过本节点；独立发布时同样必须补齐。

## 路径级准入产物

| 路径 | 必需产物 |
| --- | --- |
| 复杂需求 | 已批准的 MR、Review 报告、验证报告、发布与回滚方案 |
| 轻量需求 | 发布批次条目、验证报告、发布与回滚说明；独立发布时补充 Review 报告 |
| 快速修复 | 发布批次条目、验证报告、执行摘要中的回滚方式；独立发布时补充 Review 报告 |

## 输入

- 当前路径对应的准入产物，具体以“路径级准入产物”表为准。
- 发布窗口、灰度策略、监控指标和回滚方案。

## 执行

1. 确认构建、测试、审批、配置和环境依赖满足发布条件。
2. 准备发布说明、灰度范围、开关策略和回滚指令。
3. 按计划完成发布，并记录版本、时间、负责人和变更范围。
4. 在灰度期检查关键接口、前端异常、性能和核心业务指标。

## 本节点产物

- 发布记录。
- 灰度结果、回滚记录或发布异常说明。

## 完成条件

- 发布结果可追溯，回滚路径已验证可执行。
- 线上观察已移交给运行监控与复盘节点。

## 人工关口

发布负责人拥有最终上线、扩大灰度和回滚决定权。AI 不直接发布或合并生产分支。

## 交接

将发布记录和监控目标交给 [运行监控与复盘](./12-production-monitoring-retrospective.md)。

## 缺失处理

- 缺少验证结果、发布范围或回滚方式时，发布为 `BLOCKED`。
- 批次中任一变更没有可追溯的任务和验证结论时，不得随批次上线。
