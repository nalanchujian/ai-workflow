# 8. 开发实现

[返回工作流框架](../../AI自动化研发工作流实践指南.md)

## 产物文件

读取当前路径的实施依据，写入 `/Users/j/codes/ai-workflow/tasks/<task-id>/08-development-record.md`，并更新 `task.yaml`。

## 目标

按当前路径的已确认依据完成最小、可回滚的代码变更，并在开发阶段完成基础自测和必要文档更新。

## 适用路径

- 快速修复、轻量需求、复杂需求均在本节点完成代码实现。

## 路径级准入产物

| 路径 | 必需产物 |
| --- | --- |
| 快速修复 | 原始需求包、`path=quick_fix` 的 `change-assessment.json`、执行摘要 |
| 轻量需求 | 需求解析包、`path=lightweight` 的 `change-assessment.json`、轻量实施计划及 Owner 确认记录 |
| 复杂需求 | 已批准的实施计划、验收清单、`path=complex` 的 `change-assessment.json` |

## 输入

- 当前路径对应的准入产物，具体以“路径级准入产物”表为准。
- 接口约束、模块规范和测试环境。

## 执行

1. 使用 `$08-development-implementation` 在独立分支或 worktree 中实施，不直接修改生产分支。
2. 按既定职责边界修改页面、状态、组件、API、mapper、样式和测试。
3. 复用公共组件默认行为；如必须覆盖样式，限制在模块作用域内。
4. 执行开发自测，检查核心交互、参数、回填、异常、权限和重复操作。
5. 记录实际改动、未完成项和与当前路径依据不一致的原因。

## 本节点产物

- 代码变更集和提交记录。
- 自测结果、已知风险和必要文档更新。

## 完成条件

- 代码满足当前路径的实施依据和基础静态检查要求。
- 不包含无说明的临时 mock、调试代码或无引用旧逻辑。

## 人工关口

涉及基础层、公共组件、敏感数据或超出批准方案的改动，需要技术 Owner 再确认。

## 交接

将代码变更和自测结果交给 [质量验证](./09-quality-verification.md)。

## 缺失处理

- 当前路径缺少任一准入产物时，任务为 `BLOCKED`，不得通过口头说明或聊天记录继续开发。
- 发现需求或接口理解错误时，停止堆叠补丁，退回需求澄清或实施设计。
- 发生阻断性问题时，记录最小复现和影响范围，供前序节点重新评估。
