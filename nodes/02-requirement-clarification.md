# 2. 需求澄清

[返回工作流框架](../../AI自动化研发工作流实践指南.md)

## 产物文件

读取 `01-requirement-analysis.md`，写入 `/Users/j/codes/ai-workflow/tasks/<task-id>/02-requirement-clarification.md`，并更新 `task.yaml`。

## 目标

将自然语言需求和设计资料整理为可实施的业务规则，并明确本期范围与待确认项。

## 适用路径

- 轻量需求和复杂需求必须经过本节点。
- 快速修复仅在变更评估返回 `NEEDS_CLARIFICATION` 时进入本节点。

## 路径级准入产物

| 路径 | 必需产物 |
| --- | --- |
| 快速修复（退回澄清） | 需求解析包、变更评估中的缺失项 |
| 轻量需求 | 需求解析包 |
| 复杂需求 | 需求解析包 |

## 输入

- [需求受理](./01-requirement-intake.md)输出的需求解析包。
- 当前页面、相关代码和历史行为，必要时包含接口文档。

## 执行

1. 使用 `$02-requirement-clarification` 读取需求解析包和当前实现。
2. 提取页面入口、角色、字段、状态、权限、异常、边界条件和视觉要求。
3. 区分“已确认规则”“设计推断”“待确认问题”和“非本期范围”，并给出推荐结论。
4. 生成一次整体确认卡；需求 Owner 仅需确认整体结论或指出需要修改的项。
5. 确认后补充轻量需求的最小验收条件，不单独进入验收设计节点。

## 本节点产物

- 需求解析包。
- 待确认问题清单。
- 本期范围与排除范围。

## 完成条件

- 能清楚说明本次要解决什么、不解决什么、如何判断完成。
- 任何关键歧义都有明确的待确认记录，不以 AI 推断替代确认。

## 人工关口

产品 Owner 确认 AI 汇总的业务规则、边界和冲突处理结论。

## 交接

将需求解析包交给 [变更评估](./03-change-assessment.md)。复杂需求在评估完成后进入 [验收设计](./04-acceptance-design.md)。

## 缺失处理

- 不同截图之间存在冲突，或截图无法覆盖关键规则时，任务保持 `NEEDS_CLARIFICATION`，不得进入后续节点。
- 缺少设计稿时，可按现有设计系统给出建议，但必须标记为待确认；涉及视觉验收时不得自动通过。
