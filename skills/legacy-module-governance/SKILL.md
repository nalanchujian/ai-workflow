# 模块重构治理

在不改变业务行为的前提下重构模块。按照可独立回滚的小批次自主推进，并为每一批设置明确的质量门槛。

## 选择执行模式

优先读取用户明确提供的 `执行模式`。未明确模式时，再根据用户用词判断。

### 审查模式

- 触发方式：`执行模式：审查`，或用户只要求 Review、审查、分析、评估。
- 执行范围：完成范围确认、扫描、源码阅读、风险分级和职责边界分析。
- 输出结果：按严重程度列出已确认缺陷、架构债务和残余风险，并提供文件和行号。
- 修改权限：不修改任何文件。

### 重构模式

- 触发方式：`执行模式：重构`，或用户要求重构、实施、改造、端到端治理。
- 执行范围：完整执行本 Skill 第 1 至第 8 步，包括审查、分批修改、验证和最终 Review。
- 输出结果：说明架构结果、影响范围、验证结果和剩余风险。
- 修改权限：允许在目标模块内实施安全改动，不混入无关需求。

### 专项优化模式

- 触发方式：`执行模式：专项优化`，或用户明确要求优化某个具体问题。
- 执行范围：先确认问题及影响边界，再完成与该问题直接相关的检查、修改、验证和最终 Review。
- 输出结果：说明问题原因、修改结果、影响范围和验证结果。
- 修改权限：只修改专项问题涉及的代码，不顺带进行模块级大重构。

### 模式判断优先级

1. 用户明确指定的 `执行模式` 优先级最高。
2. “只审查”“不要修改”等限制词强制进入审查模式。
3. “审查并重构”“分析后实施”等请求进入重构模式，审查只是重构的前置步骤。
4. “优化重复请求”“拆分某个 Controller”等有明确目标的请求进入专项优化模式。
5. 无法判断且修改可能产生风险时，默认按审查模式执行。

### 附加执行策略

- 用户要求自主交付时，在所选模式范围内持续完成所有未被阻塞的批次。只有涉及业务语义、安全、数据完整性、公共 API 兼容性或生产环境状态时才询问用户。
- 未获得用户请求或项目固定规则授权时，不提交、不推送、不合并、不发布，也不创建 PR。

推荐输入格式：

```text
使用 $legacy-module-governance
执行模式：审查 / 重构 / 专项优化
目标模块：<module-path>
重点关注：<可选>
执行授权：<是否允许修改、提交、推送>
```

## 按需读取资料

- 审查或修改模块前，读取 [references/baseline.md](references/baseline.md)。
- 制定批次或处理不明确决策前，读取 [references/risk-and-decisions.md](references/risk-and-decisions.md)。
- 设计职责边界、目录结构或状态归属时，读取 [references/links-reference.md](references/links-reference.md)。把 Links 当作参考，不要机械复制结构。
- 模块缺少文档时，使用 `assets/module-docs/` 下的模板。

## 执行流程

### 1. 确认范围和仓库状态

1. 找到模块根目录、路由入口、包管理器、构建命令、负责人和相关公共服务。
2. 修改前检查 Git 状态，保留用户的无关改动。
3. 根据 `references/risk-and-decisions.md` 判断模块层级和改动风险。
4. 明确重构前后必须保持一致的核心业务链路。

### 2. 先审查，再修改

目标是本地源码目录时，先执行本 Skill 根目录下的扫描脚本。脚本路径相对于 `SKILL.md` 解析，不要从项目工作目录查找：

```bash
scripts/audit_module.sh <module-path>
```

然后直接阅读源码并确认：

- entry、page、layout、component、hook、API、mapper、shared 和 style 的职责边界；
- 列表、筛选、表单、弹窗、分页和刷新状态分别由谁管理；
- 是否存在组件直接请求接口、全局 service 耦合、重复请求和并发覆盖；
- 权限、文案、日期/时区、状态映射和功能开关是否统一；
- 是否覆盖公共组件样式、使用全局选择器、固定尺寸或复杂滚动逻辑；
- 是否残留死代码、旧流程、mock、临时开关、兼容别名和过期文档；
- 是否存在超大文件或同时承担多个职责的 Controller。

审查模式下，按严重程度输出问题，并提供文件和行号。区分已确认缺陷、架构债务和残余风险。

### 3. 明确目标职责边界

修改前说明各层归属：

- entry/page/layout：页面装配和路由级决策；
- components：视图渲染和事件透传；
- hooks/controllers：业务状态和请求编排；
- APIs：接口地址、请求参数和响应类型；
- mappers/utils：纯数据转换和兼容规则；
- shared：多个子业务稳定复用的能力；
- styles：局部展示样式，不包含业务判断。

不要创建模块不需要的目录。目标是让一个需求的核心实现能在一到两个文件内定位，而不是追求文件数量。

### 4. 制定可回滚的批次

默认按以下顺序执行，已完成或不适用的步骤可以跳过：

1. 新增或更新模块文档。
2. 从业务逻辑中抽离页面装配层。
3. 分离列表/查询状态和弹窗/表单状态。
4. 收口 API、请求构造、响应类型、mapper、状态码和文案。
5. 按稳定职责拆分超大 Controller。
6. 去除重复请求，并防止旧响应覆盖新状态。
7. 清理死代码、mock、过期兼容和危险样式覆盖。
8. 业务稳定后再优化派生数据和渲染依赖。

等价重构、接口契约变化和 UI 变化要分批处理，不要在同一批混入无关修复。

### 5. 自主实施

- 修改前阅读目标文件及其调用方。
- 保持 API 行为、权限、校验、文案、功能开关、埋点和 UI 状态一致。
- 使用明确的 adapter，避免兼容判断散落在组件中。
- 后端字段、表单字段、展示模型和提交参数不一致时，分别定义类型。
- 分离列表状态和弹窗状态，避免弹窗输入拖动整张列表重渲染。
- 明确共享筛选状态，只向每个 Tab 暴露允许读写的字段。
- 优先复用组件默认样式，必要覆盖必须限制在模块内部。
- 只有证明代码无引用，或符合已批准的下线规则时才能删除。
- 某一项被阻塞时，记录原因并继续其他安全批次。

### 6. 验证每个批次

执行仓库中可用的最强验证：

1. 对修改文件执行 lint；
2. 执行类型检查或针对性类型检查；
3. 执行相关单元测试或集成测试；
4. 风险匹配时执行构建；
5. 对关键交互执行浏览器或应用冒烟测试；
6. 执行 `git diff --check` 并审查最终 diff。

根据改动范围验证初始化、搜索、清空、分页、Tab 切换、新建、编辑、删除、权限、错误态、空状态和重复操作。

如果全量检查被项目历史问题阻塞，不得声称全量通过。必须区分已有失败和本次新增失败。

### 7. 独立执行最终 Review

把最终 diff 当作其他工程师提交的代码重新审查，重点检查：

- 是否改变业务行为；
- 是否新增重复请求或漏请求；
- 是否存在闭包过期、定时器未清理、竞态和响应顺序问题；
- 表单 value/label、字符串/数字类型是否不匹配；
- 权限和功能开关是否回退；
- 是否意外影响公共样式；
- 是否存在漏导出、死 import、过期文档和临时代码。

交付前修复所有已确认问题。

### 8. 输出结果

最终说明保持简洁，包括：

- 用户可见结果或架构结果；
- 影响范围；
- 已执行的验证；
- 剩余阻塞或风险；
- 只有获得授权并实际执行后，才说明提交、推送或 PR 状态。

## 完成门槛

满足以下条件后，模块重构才算完成：

- 职责边界和状态归属清晰；
- 组件不会绕过约定的 API 边界；
- 列表、表单、弹窗和共享状态不存在隐藏耦合；
- 不存在已确认的重复请求和旧响应覆盖问题；
- 权限、状态、文案、日期和样式都有单一可追踪来源；
- 旧代码已删除或有明确下线条件；
- 模块文档与当前实现一致；
- 修改文件通过验证，并明确披露项目已有失败。

如果某个门槛无法安全满足，只停止受影响的批次，并说明准确的阻塞原因。
