# 风险与自主决策规则

## 1. 模块分层

修改前判断模块层级：

- A 级：收入、认证、权限、支付、发布、破坏性操作或高频核心业务。
- B 级：重要运营流程，失败可恢复且经常迭代。
- C 级：辅助、报表、设置或低频维护流程。
- D 级：已废弃、实验、演示或计划下线代码。

模块越重要，批次越小，验证要求越高。

## 2. 改动风险

- P0：生产安全、信息安全、数据丢失、认证或不可逆外部操作。不得自行推断。
- P1：公共 API、业务契约、权限语义、金额计算或跨模块共享行为。必须有明确证据或既定规则。
- P2：模块内部架构、状态归属、请求去重、mapper 收口和组件边界。可以分批自主处理。
- P3：注释、文档、死 import、局部命名和已证明无引用的代码。可以自主处理。

## 3. 默认决策

用户不参与时：

- 保持当前业务结果和交互不变。
- 优先增加 adapter 或 wrapper，不优先修改公共基础设施。
- 不明确的逻辑保持原样，并记录为剩余技术债。
- 不确定是否可删除时跳过删除，继续其他安全工作。
- 保持服务端契约和功能开关行为不变。
- 没有设计或需求依据时，保留组件默认行为。
- 优先复用项目内已维护且安全的模式。
- 选择能建立清晰边界的最小改动。
- 未获授权且项目未使用时，不新增依赖。
- 不修改工作区中的无关改动。

## 4. 必须询问的场景

只有穷尽安全排查手段，并且决策会产生以下影响时才询问用户：

- 改变业务含义或用户规则；
- 改变权限、安全、金额或数据保留行为；
- 删除公共或外部调用的契约；
- 需要生产凭证或外部团队协作；
- 会覆盖用户正在进行且存在冲突的改动；
- 会执行不可逆外部操作；
- 必须在明显不同的产品体验中选择一种。

其他情况选择保守方案，说明假设并继续执行。

## 5. 交付门槛

- 不能仅因为编译通过就合并或发布。
- A 级模块的每条关键改动链路都要有针对性运行验证。
- B 级模块至少需要修改文件 lint、类型/构建证据和冒烟验证。
- 无法搭建运行环境时，C/D 级模块可以使用针对性静态检查。
- 项目全量检查已有失败时，要证明本次文件没有新增失败并披露基线。
- 某个门槛失败时，只阻塞受影响批次；除非它使整个重构结论失效。
