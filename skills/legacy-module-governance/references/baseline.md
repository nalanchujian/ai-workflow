# 模块重构基线

## 文档目录

## 1. 结构

- 路由和页面入口只负责装配。
- 独立子业务必须有自己的目录。
- 按需使用 `apis`、`components`、`hooks`、`layouts`、`shared`、`styles`，不要为了形式创建空目录。
- 视图组件不处理接口地址和请求编排。
- 纯数据转换放在 React 组件和 Hook 之外。
- 一个需求的核心实现应能在一到两个可预期文件中定位。
- 不要只为减少行数拆文件，只有职责或变化原因不同时才拆分。

## 2. 状态与请求

- 筛选、列表、分页、选择项、弹窗、表单和选项状态都要有唯一负责人。
- 列表状态与弹窗/抽屉状态分离。
- 明确跨 Tab 共享字段，不要仅靠约定共享无限制 Form 或 Store。
- 集中构造请求参数。
- 统一第一页刷新和当前页刷新入口。
- 去重同参数请求，并防止旧异步响应覆盖新状态。
- 卸载时清理定时器、订阅和未执行的防抖任务。
- 明确轮询、延迟刷新由谁管理，以及何时停止。
- 除非操作明确要求，否则刷新时保留当前分页。

## 3. 表单与列表

- 显式实现校验；除非产品明确要求，按钮禁用不能代替校验。
- 新建和编辑使用一致的校验交互。
- 下拉编辑回填必须展示 label，不能展示原始 ID。
- 在 adapter 边界统一字符串和数字 ID。
- 懒加载下拉覆盖初始化、编辑回填、搜索、清空、分页和选项删除场景。
- 表格列配置不要堆在页面组件中。
- 复杂 Cell 和 Actions 应独立拆分。
- 分页只能由一个状态源控制。
- 分别建模加载、空数据、错误、断连和无权限状态。

## 4. API 与数据映射

- 模块请求统一经过模块 API 层。
- 组件不关心接口地址和传输细节。
- 列表、详情、表单和提交参数不一致时分别定义类型。
- 使用 mapper 完成后端到展示、表单到提交的转换。
- 兼容别名统一放在 mapper，并说明保留原因。
- 后端契约稳定后及时删除旧别名。
- 状态码、用户文案、日期/时区和展示标签统一映射。
- 不要自行编造后端缺失字段的值。

## 5. 权限、文案与样式

- 记录权限名称、作用范围、交互方式和代码位置。
- 区分隐藏、展示但禁用、可点击后由服务端校验。
- 除非需求不同，新旧入口使用同一权限。
- Toast 和字段错误文案统一收口。
- 明确服务端 message 是透传还是替换。
- 优先使用组件库默认样式。
- 必要的样式覆盖必须限制在模块根节点下。
- 非必要不依赖组件内部 DOM；确实需要时记录原因。
- 统一模块内间距、按钮、Alert、Tooltip、Table、Scrollbar 和状态样式。

## 6. 性能与遗留代码

- 拆分同时管理多个无关领域的 Controller。
- 只缓存有意义的派生集合和稳定 adapter，不缓存简单表达式。
- 弹窗输入不能导致列表行和列整体重渲染。
- 不增加没有明确失效责任的缓存层。
- 搜索死导出、旧组件、旧接口、mock、功能开关和临时值。
- 隔离 Legacy 代码，并标明负责人和下线条件。
- 功能开关或权限仍可能选择旧流程时，不得删除旧流程。

## 7. 文档与验收

复杂模块至少维护：

- `README.md`：模块职责、结构和阅读顺序；
- `FLOW.md`：初始化、查询、提交、刷新和状态流；
- `API.md`：接口、参数、响应、状态码和调用方；
- `PERMISSIONS.md`：权限矩阵和 UI 行为。

验收时应能快速回答：

- 页面入口在哪里？
- 谁管理列表状态和表单状态？
- 请求和提交参数在哪里构造？
- 后端字段在哪里映射？
- 权限和状态文案在哪里定义？
- 新需求能否不搜索整个仓库就定位？
- 每个批次能否独立回滚？
