# 需求受理

将产品需求和设计资料收敛为后续开发、审查或测试可直接使用的输入。默认不修改业务代码；只有用户明确要求实施时，才把本次结果交给后续开发流程。

## 截图优先策略

不接收 Lark 或 Figma 链接。每次解析只接收同一条消息中上传的两张截图：第 1 张为需求文档截图，第 2 张为设计稿截图。

截图通过当前对话直接上传，不需要 HTTP 链接、文件路径或外部地址。需求、设计、接口和交互事实仅从这两张截图提取；接口信息未在需求截图中出现时，必须标记为待确认。

截图不足以确认字段、状态、权限、边界或精确文案时，列为待确认，并说明需要补充的页面、区域或状态截图。

按以下优先级确认事实：

1. 本轮消息第 1 张需求文档截图和第 2 张设计稿截图。
2. 当前仓库实现。

截图可作为业务和视觉依据，但无法从截图确认的接口、状态机、权限或边界条件必须标记为待确认，不能自行补全为事实。

## 执行流程

### 1. 收集资料

- 从截图提取需求入口、目标模块、用户角色、主流程、异常流程、状态、权限、接口和验收要求。
- 从设计截图提取目标区域的布局、颜色、尺寸、组件状态和交互说明；截图无法覆盖的信息必须记录为待确认。
- 阅读当前实现前，先定位路由入口、页面装配层、相关 API、状态管理和公共组件。

### 2. 建立需求事实表

将每条需求归类为：

- 页面与入口
- 字段与表单校验
- 列表、筛选、分页和空状态
- 操作、状态流和错误处理
- 权限、功能开关和埋点
- API、数据字段和日期/时区
- 视觉与交互

每条写清楚来源、当前实现、目标行为和结论。区分：`已确认`、`需确认`、`与现状不一致`、`超出前端范围`。

### 3. 比对 Figma 与当前页面

- 先比较信息结构和交互，再比较视觉细节。
- 视觉比较至少覆盖：页面边距、区块间距、字号/字重、颜色、边框、圆角、按钮、表格、弹窗、空/禁用/错误状态。
- 不把账号数据、时区、环境域名等运行时差异误判为设计问题。
- 发现公共组件默认样式被覆盖时，单列风险，不建议为了单页效果直接改公共组件。

### 4. 扫描实现影响范围

在生成开发计划前完成影响范围扫描：

- 找到路由、入口 Tab、功能开关和权限点。
- 找到 API 调用、类型、mapper、状态码和文案来源。
- 找到列表、表单、弹窗、共享筛选和刷新链路。
- 找到可复用公共组件、旧入口、旧接口、mock 和潜在兼容逻辑。
- 标记可能影响其他模块或存在重复请求、状态耦合、样式污染的区域。

### 5. 写入固定交接物

按 [references/output-template.md](references/output-template.md) 生成以下固定交接物：

- 结论与已确认范围。
- 截图覆盖范围和缺失信息。
- 需求事实表。
- 设计截图与当前页面差异。
- 初步影响范围。
- 待确认项。
- `需求解析包`。

每次执行必须创建一个独立任务目录，并把完整结果写入：

`/Users/j/codes/ai-workflow/tasks/<task-id>/01-requirement-analysis.md`

同时创建或更新同目录的 `task.yaml`，至少记录 `task_id`、`target_module`、当前节点状态和产物文件名。`task-id` 默认使用 `<YYYYMMDD-HHMMSS>-<模块名>`；若当前任务已提供任务目录，则复用该目录，不能另建平行任务目录。

聊天中只摘要结论并给出产物路径。`01-requirement-analysis.md` 是交给
`$04-acceptance-design`、开发计划或重构流程的唯一标准输入，避免下一阶段重复读取同一份需求或依赖聊天上下文。

不接收“输出目标”参数；如果截图或目标模块不足以形成交接物，固定输出待确认项并停止在需求受理节点。

按 [references/handoff-template.md](references/handoff-template.md) 生成该解析包。除非用户明确要求其他位置，否则统一写入上述任务目录；不再只在当前对话输出。

## 质量门槛

- 不把设计截图中的静态视觉稿推断为完整业务规则。
- 不把后端职责误列为前端实现项；仅标记前端需要消费的字段或状态。
- 每个已确认差异要能定位到截图或现有代码位置。
- 每个待确认项都要说明它阻塞哪项实现或验收。
- 测试项必须可操作、可观察，避免只写“验证正常”。


完成后告知 `01-requirement-analysis.md` 的绝对路径；后续节点直接读取该文件。
