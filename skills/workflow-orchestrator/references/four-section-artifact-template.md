# Markdown 节点产物结构

适用于节点注册表中输出为 `.md` 的 7 个节点；`requirement-routing.json` 继续使用 JSON Schema。旧产物没有 `template_version` 时按原结构读取，新产物固定使用版本 2。

```yaml
---
task_id: <task-id>
node: <node-id>
status: completed | blocked
attempt: 1
template_version: 2
---
```

正文只使用以下 4 个一级标题，顺序固定且每个只出现一次。节点专属栏目使用二级标题，并以对应节点的输出模板为准。使用 `#` 与 `##`，让两个层级在渲染时有明显的字号差异。

```markdown
# 结果

<让 Owner 在 30 秒内看懂本节点做出了什么结论、覆盖了多少内容>

# 交付

## <节点专属栏目>

<后续节点真正需要使用的事实、用例、范围、方案或证据>

# 未决项

| ID | 类型 | 问题 | 影响 | Owner | 关闭条件 | 重试节点 |
| --- | --- | --- | --- | --- | --- | --- |

# 交接

| 项目 | 内容 |
| --- | --- |
| 交付给 | <下一节点或材料 Owner> |
| 本节点提供 | <可直接复用的输入> |
| 准入条件 | <开始下一节点前必须满足的条件> |
| Gate 含义 | <批准本节点具体代表什么；blocked 时写“不创建 Gate”> |
```

约束：

- `结果` 只写结论、覆盖数量和状态，不重复正文。
- `交付` 只保存本节点负责生产的内容，不提前代做下游节点。
- `未决项` 只保存仍需处理的问题；没有时写“无”。`status: blocked` 时至少有一个阻塞项，并给出 Owner、关闭条件和重试节点。
- `交接` 只说明责任转移，不复制 `task.yaml` 中的运行状态。Gate、attempt 和下一节点仍只由 `task-state.rb` 管理。
