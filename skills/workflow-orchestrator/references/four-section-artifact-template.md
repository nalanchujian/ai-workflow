# Markdown 节点产物结构

适用于输出为 Markdown 的 7 个节点；`requirement-routing.json` 继续使用 JSON Schema。所有新产物统一使用 `artifact_schema_version: 1`；旧产物的 `template_version: 2/3` 继续兼容。

```yaml
---
task_id: <task-id>
node: <node-id>
status: completed | blocked
attempt: 1
artifact_schema_version: 1
---
```

正文只保留 4 个一级标题，不使用二级标题。节点内部栏目用粗体标签，避免依赖 Markdown 渲染器不明显的标题字号差异。

```markdown
# 1. 结果

> **状态：** <完成 / 阻塞>
> **结论：** <一句话结论>
> **规模：** <本节点实际产出数量>

# 2. 产出

**<节点专属栏目>**

<表格、清单或证据>

# 3. 待确认

<仍需处理的问题；没有则写“无”>

# 4. 下一步

- 当前动作：
- 完成条件：
```

约束：

- “结果”只写状态、结论和规模，不复制正文。
- “产出”只保存本节点新增的事实、判断或证据；通过上游 ID 建立追踪关系，不复制前序节点原文。栏目用单独一行粗体标签，不创建二级标题。
- “待确认”不重复“产出”。需求节点在这里合并问题与推荐决策；其他 Markdown 节点统一使用“ID、类型、事项与影响、Owner、责任节点、完成条件”六列，只记录仍未关闭的风险、缺陷或 blocker。
- “下一步”只保留“当前动作”和“完成条件”。后续节点与 Gate 指令由 `task-state.rb` 实时生成并在聊天中展示，不写入产物。
- 每份产物最多 3 张表，每张表最多 6 列；没有内容的表直接写“无”，单元格只写短句，详细日志保存为独立证据。
- 需求受理的未登记草稿可以使用 `status: awaiting_confirmation`；它表示正常等待逐项确认。`status: blocked` 只表示缺少资料、存在缺陷或无法继续执行。
