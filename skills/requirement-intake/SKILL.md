# 需求受理与确认

任务目录统一写作 `<tasks-root>/<task-id>`；只读取 `task.yaml` 校验准入，不得直接创建、修改或替换它。节点只原子写入本节点产物并向 `$workflow-orchestrator` 报告结果；普通状态由 `workflow-orchestrator` actor 推进，人工 Gate 仅在用户原样提交精确确认指令后由 `workflow-owner` actor 决策，状态处理遵循 [任务生命周期指南](../workflow-orchestrator/references/guides/task-lifecycle.md)。产物结构必须符合 [节点产物 Schema](../workflow-orchestrator/references/contracts/node-artifact.schema.json)，Markdown 正文遵循 [统一产物结构](../workflow-orchestrator/references/four-section-artifact-template.md)和[本节点输出模板](references/output-template.md)，重跑时遵循 [节点重跑指南](../workflow-orchestrator/references/guides/node-rerun.md)。

将产品需求和设计资料收敛为后续节点需要的最小事实输入，并在同一节点内逐条确认资料无法直接证明的疑点。默认不修改业务代码；只有用户明确要求实施时，才把本次结果交给后续开发流程。

本节点分为两个连续阶段：先回答“资料证明了什么、当前实现是什么、还有哪些疑点”，再通过 Owner 对话回答“每个疑点最终采用什么规则”。需求事实使用 `RF-*`，疑点使用 `RQ-*`，确认决策使用 `CL-*`；三者必须分区维护，不能把推荐答案改写成原始事实。

## 资料识别

不要求用户声明资料类型，也不在产物中记录分类字段。直接阅读用户提供的需求图片、设计图片、文字、日志、接口契约和仓库证据，并按内容提取：

- 业务目标、规则和验收要求。
- 页面布局、组件状态、交互和视觉要求。
- 缺陷复现步骤、预期行为、实际行为和失败证据。
- 接口字段、兼容要求、调用方和错误处理要求。
- 技术目标、约束、非目标和影响范围。

一份资料可以同时包含多类内容，不强制选择唯一类别。仅在图片或文字无法确认其含义，且该含义会阻塞范围判断时，输出待澄清项并停止。Lark 或 Figma 链接不能作为唯一事实来源；无法读取时要求截图或导出的可访问内容。

## 资料优先级

截图通过当前对话直接上传。需求、设计、接口和交互事实仅从提供资料提取；接口信息未在资料中出现时，必须标记为待澄清。

截图不足以确认字段、状态、权限、边界或精确文案时，列为待澄清，并说明需要补充的页面、区域或状态截图。

按以下优先级确认事实：

1. 用户提供的需求图片、设计图片、文字和其他直接证据。
2. 当前仓库实现、可复现测试和运行证据。

截图可作为业务和视觉依据，但无法从截图确认的接口、状态机、权限或边界条件必须标记为待澄清，不能自行补全为事实。

## 执行流程

### 1. 收集资料

- 逐份识别图片和文字表达的需求事实，从中提取入口、目标模块、用户角色、主流程、异常流程、状态、权限、接口和验收要求。
- 资料包含界面或视觉信息时，提取布局、颜色、尺寸、组件状态和交互说明；资料无法覆盖的信息必须记录为待澄清。
- 阅读当前实现时只定位目标模块根目录、路由入口和页面装配层，用于确认需求对应的真实代码位置。

### 2. 建立需求事实

先按用户可感知的业务模块分组，例如页面、功能区或公共规则：

- 页面与入口
- 字段与表单校验
- 列表、筛选、分页和空状态
- 操作、状态流和错误处理
- 权限、功能开关和埋点
- API、数据字段和日期/时区
- 视觉与交互

`# 交付` 下的 `## 需求事实` 使用一张表格，列固定为“事实 ID、模块、目标行为、当前差异、来源、状态”。模块表示页面、功能区或公共规则，目标行为表示模块内可观察的单项能力。`RF-*` 在整份产物中连续递增；状态区分：`已确认`、`需确认`、`超出范围`。

每行只记录一个具体行为；同一模块可以连续出现多行，以“模块 → 具体行为”的列关系表达层级。跨模块规则统一归入“公共规则”，不要在每个模块中重复登记。

业务规则、视觉要求及其与当前实现的差异统一写入需求事实，不再维护独立差异表。事实只记录证据，不给出最终业务口径、接口字段、实现方式或方案选择；截图和代码无法共同证明的内容不得通过“建议处理”补成事实。

### 3. 比对目标行为与当前实现

- 将预期行为与当前实现或可复现证据直接写入对应 `RF-*`。
- 资料包含界面或视觉要求时，事实至少覆盖有明确证据的信息结构、交互、页面间距、文字、颜色、边框、按钮、表格、弹窗和组件状态。
- 不把账号数据、时区、环境域名等运行时差异误判为设计问题。
- 发现公共组件默认样式被覆盖时，单列风险，不建议为了单页效果直接改公共组件。

### 4. 记录分流线索

只生成后续分流需要的初步代码线索，并在产物中明确标记为“分流线索”：

- 定位目标模块根目录、路由或页面入口以及直接装配文件。
- 记录从入口即可观察到的权限、功能开关、外部接口或公共能力线索。
- 标记“可能跨模块”或“可能涉及公共能力”，但不在本节点追踪完整调用链。

不要枚举所有受影响文件，不分析完整请求链、状态流、并发和回归范围。需求分流只验证这些线索是否触发复杂路径；完整影响分析只由影响分析节点负责。

### 5. 写入固定交接物

按 [references/output-template.md](references/output-template.md) 生成版本 2 交接物；节点专属表格全部放入“交付”：

- 结论。
- 带 `RF-*` 编号的需求事实。
- 分流线索。
- 带 `RQ-*` 编号的待澄清项。
- 带 `CL-*` 编号的确认决策。

每个待澄清项使用 `RQ-001`、`RQ-002` 递增，并引用相关 `RF-*`。问题只描述证据缺口、冲突和阻塞内容，不在问题表中填写推荐结论。没有待澄清项时明确写“无”，确认决策同样写“无”。

### 6. 逐条确认疑点

1. 严格按编号顺序处理，一轮对话只展示一个 `RQ-*`。展示前读取 [当前问题对话模板](references/question-dialog-template.md)，每轮按该模板完整展示疑点编号、疑点标题、具体问题、产生原因、推荐答案、影响范围和有效回复格式，不能提前展示或默认确认后续问题。对话模板只定义聊天格式，不写入节点产物。
2. 当前问题只接受两类 Owner 回复：`确认 RQ-001` 表示接受本轮全部推荐答案；`修改 RQ-001：<最终规则>` 表示用 Owner 给出的规则替换推荐答案并确认。编号必须与当前问题完全一致，“可以”“继续”等模糊回复不能确认。
3. 每个 `RQ-*` 至少关联一个 `CL-*`。一个问题可拆成多个决策，一个决策也可引用多个问题；确认或修改后将相关 `CL-*` 状态写为“已人工确认”，再展示下一问题。
4. 对话进行中只将进度保存在尚未登记的 `requirement-analysis.md` 草稿的 `RQ/CL` 表中，frontmatter 使用 `status: blocked`；不得调用 `record-result`，任务保持本节点执行中。恢复时按 `RQ-*` 编号顺序查找首个采用规则为空、缺少关联 `CL-*` 或存在关联 `CL-*` 状态不是“已人工确认”的问题，并从该问题继续；不创建额外进度文件。
5. 只有全部 `RQ-*` 都有非空采用规则且相关 `CL-*` 均为“已人工确认”时，才能把草稿改为 `status: completed` 并报告编排器。状态 CLI 会强制校验完整映射、未知引用和逐条确认状态。

每次执行必须创建一个独立任务目录，并把完整结果写入：

`<tasks-root>/<task-id>/requirement-analysis.md`

新任务目录与 `task.yaml` 由编排器先通过 `task-state.rb init` 创建。全部疑点逐条确认后，报告资料覆盖、`RQ-* -> CL-*` 关系和最终决策，请求 `workflow-owner` 通过本节点 Gate 确认汇总产物。Gate 批准后下一节点固定为 `requirement-routing`。`task-id` 默认使用 `<YYYYMMDD-HHMMSS>-<模块名>`；若当前任务已提供任务目录，则复用该目录，不能另建平行任务目录。

逐条对话期间，聊天只展示当前疑点及两种有效回复；最终登记后再摘要结论并给出产物路径。`requirement-analysis.md` 是需求分流的已确认需求输入，但不是完整影响分析或实施方案。

不接收“输出目标”参数；如果当前资料或目标模块不足以形成交接物，固定输出待澄清项并停止在需求受理节点。

## 质量门槛

- 不把设计截图中的静态视觉稿推断为完整业务规则。
- 不把后端职责误列为前端实现项；仅标记前端需要消费的字段或状态。
- 每个事实都要能定位到资料或现有代码位置。
- 每个事实必须填写明确的业务模块和具体行为。
- 每个待澄清项都要说明它阻塞哪项实现或验收。
- 事实和待澄清项分别使用 `RF-*`、`RQ-*`，待澄清项必须引用来源事实。
- 推荐规则只能写入 `CL-*` 决策，不能混入 `RF-*` 事实。
- 不在本节点设计接口参数名、存储算法、组件结构、样式实现或测试步骤。
- 未覆盖全部 `RQ-*`、存在非“已人工确认”决策或未获最终 Gate 批准时，不得进入需求分流。

## 推荐调用方式

```text
使用 $requirement-intake
需求资料：<直接上传需求图片、设计图片或补充文字>
目标模块：<本地路径或业务名称>
```

```text
使用 $requirement-intake
缺陷：<复现步骤、预期行为、实际行为>
目标模块：<本地路径或业务名称>
```

存在疑点时先逐条对话；全部确认后告知 `requirement-analysis.md` 的绝对路径并等待最终 Gate，需求分流只在该 Gate 批准后读取此文件。
